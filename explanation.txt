# Curated School Datamodel Agent — Architecture & Explanation

This document serves as an overview for any future AI developer or agent working on this codebase.

## 1. Project Purpose
The project is a natural-language-to-SQL (NL2SQL) system that allows users to query a curated
school database schema (focusing on student demographics, risk metrics, attendance, academic
performance, meals, schemes, and infrastructure). It serves both a dashboard frontend (via APIs)
and an interactive AI agent chat.

## 2. Core Stack
*   **Web Framework**: FastAPI (in `app.py`) serves the dashboard web pages, statistics endpoints
    (`/api/stats`), list endpoints (`/api/students`), and the agent chat endpoints
    (`/api/chat/*`, `/ask`).
*   **Agent Logic**: LangGraph (in `my_agent/agent.py` and `my_agent/utils/nodes.py`) implements
    a stateful agent.
*   **Local LLM Integration**: Ollama (via `langchain-ollama`) runs the chat model
    (e.g., `qwen3.5:2b` for lightweight memory setups or `qwen3.5:9b`) and the embedding model
    (`nomic-embed-text`).
*   **MCP Subprocesses**: Two Model Context Protocol (MCP) servers run in the background
    (managed via `my_agent/utils/tools.py`):
    1.  **RAG Server** (`MCP/mcp_rag.py`): Semantic retrieval of schema DDLs from a local
        Milvus Lite collection (`MCP/milvus_schemas.db`).
    2.  **SQL Server**: Handles SQL execution. Depending on configuration, it loads either
        `mcp_sql_execution.py` (for SQLite) or `mcp_hive_execution.py` (for Hive).

## 3. Vector Injection Pipeline (`pipeline.py`)

`pipeline.py` is a standalone, config-driven pipeline that populates the Milvus collection
used by the RAG MCP server.  It is run once (or re-run to refresh) before starting the app.

### Milvus Collection Layout
One collection (`schema_chunks`) with two named partitions:

    Partition         Contents               embedding_text     raw_ddl
    ──────────────    ───────────────────    ──────────────     ──────────
    schema_store      One doc per YAML       Schema prose       DDL string
    few_shot_store    NL→SQL exemplars       NL question        Gold SQL

### Milvus Schema (7 fields, shared by both partitions)
    id             VARCHAR(64)     Primary key (UUID)
    database_name  VARCHAR(256)    DB name  / few-shot: use_case
    table_name     VARCHAR(256)    Table    / few-shot: intent
    embedding_text VARCHAR(10000)  Embedded text (schema prose or NL question)
    raw_ddl        VARCHAR(65535)  DDL string or gold SQL answer
    source_file    VARCHAR(512)    Path to source .yaml or .jsonl
    embedding      FLOAT_VECTOR    768-dim vector (nomic-embed-text)

### Running the Pipeline
    # Inject table schemas
    python pipeline.py --config config.yaml

    # Inject few-shot NL→SQL pairs
    python pipeline.py --config config.yaml --fewshots school_dropout_fewshots_200.jsonl

    # Override YAML directory
    python pipeline.py --config config.yaml --yaml_dir ./schema/curated_datamodels/tables

### Pipeline Config (`config.yaml`)
    yaml_dir: ./schema/curated_datamodels/tables
    embedding:
      provider: ollama
      model: nomic-embed-text
      ollama_url: http://localhost:11434/api/embeddings
    vector_db:
      provider: milvus
      milvus:
        uri: ./milvus_schemas.db
        collection: schema_chunks
        dim: 768
        metric_type: COSINE

### Supported Embedding Providers
    provider: ollama              → Ollama local server (default)
    provider: openai              → OpenAI API (requires OPENAI_API_KEY env var)
    provider: sentence_transformers → Local HuggingFace model (no API key)

### Few-Shot JSONL Format
Each line must have at minimum `id`, `question`, `sql`. Optional fields `use_case` and
`intent` are mapped to `database_name` and `table_name` for downstream filtering.

### Join Relations Processing (join_relations.yaml)
Although the table-specific YAML files are loaded by the primary schema injection pipeline, the explicit join mappings are defined in [join_relations.yaml](file:///Users/prakhar/Downloads/Student_Dropout_v1.0/schema/curated_datamodels/joins/join_relations.yaml). Here is how they are processed and retrieved:

1. **Indexing/Setup Phase:**
   * When setting up the vector index (using [build_milvus_index.py](file:///Users/prakhar/Downloads/Student_Dropout_v1.0/MCP/build_milvus_index.py)), the function `load_join_chunks()` parses [join_relations.yaml](file:///Users/prakhar/Downloads/Student_Dropout_v1.0/schema/curated_datamodels/joins/join_relations.yaml).
   * It extracts each defined join relation (e.g. `from_table`, `from_column`, `to_table`, `to_column`, `cardinality`, `usage`) and formats it as descriptive text:
     `{name}: {from_table}.{from_column} -> {to_table}.{to_column} ({cardinality}). Usage: {usage}`.
   * All join descriptions are aggregated into a single text block with the metadata attribute `table_name` set to `"__join_relations__"`.
   * This chunk is vectorized using the configured embedding model (e.g. `nomic-embed-text`) and inserted into the `schema_store` partition of the Milvus Lite collection ([milvus_schemas.db](file:///Users/prakhar/Downloads/Student_Dropout_v1.0/MCP/milvus_schemas.db)).

2. **Query Phase (Runtime):**
   * When a user submits a query, the LangGraph agent calls `retrive_schema_rag(query, top_k)` via the schema RAG MCP server ([mcp_rag.py](file:///Users/prakhar/Downloads/Student_Dropout_v1.0/MCP/mcp_rag.py)).
   * The RAG MCP server embeds the query and searches the `schema_store` partition.
   * If the user's query pertains to relationships, joins, or mentions multiple tables, the semantic search retrieves the chunk with `table_name: "__join_relations__"`.
   * The retrieved join details are passed back to the LangGraph agent, providing the necessary context for the LLM to write correct SQL table joins.

## 4. Impala Execution Layer (`MCP/`)

When `HIVE_MCP_ENABLED=true`, the agent executes SQL against Impala on CDP using
Impyla with Kerberos (GSSAPI) authentication.  All execution-layer code lives in `MCP/`.

### Files

    MCP/hive_config.yaml         All infrastructure values (host, port, timeout).
                                 Uses `query_engine:` as the canonical section.
                                 Nothing is hardcoded in Python.

    MCP/hive_executor.py         HiveExecutor class — the core execution engine
                                 (name preserved for import compatibility):
                                   • Impyla connect with GSSAPI/Kerberos auth
                                   • Lazy connect + auto-reconnect
                                   • SQL validation (SELECT-only guard)
                                   • Query timeout via threading + cursor.cancel()
                                   • Structured JSON result (same contract as SQLite MCP)
                                 No PyHive code, type patches, or timestamp workarounds.
                                 Impyla returns native datetime.date / datetime.datetime.

    MCP/hive_startup_check.py    7-step pre-flight validator. Run before app start:
                                   python MCP/hive_startup_check.py
                                 Checks:
                                   1. JAVA_HOME exists
                                   2. HADOOP_HOME + hadoop binary present
                                   3. HADOOP_CONF_DIR has core-site.xml, hdfs-site.xml
                                   4. Kerberos ticket valid (klist)
                                   5. HDFS reachable (hdfs dfs -ls /)
                                   6. Impala TCP port open (21050)
                                   7. Impala connectivity: SELECT 1, SHOW DATABASES,
                                      and Iceberg table read validation

    MCP/mcp_hive_execution.py    MCP server — wraps HiveExecutor as the execute_sql
                                 tool. Guarded by HIVE_MCP_ENABLED; returns a clear
                                 error when disabled (safe for local SQLite dev).

### Iceberg Compatibility

Iceberg tables were previously accessed through Hive, which caused runtime failures:

    java.lang.NoSuchMethodError: org.apache.iceberg.parquet.ParquetSchemaUtil.hasIds(...)

Root cause: Iceberg/Parquet JAR version skew in the Hive vectorized execution path.
Resolution: Impala accesses the same Iceberg tables without this issue — no client-side
workarounds required.

### Runtime Environment (server only)

The following must be set in the shell before starting the app with Impala enabled:

    export JAVA_HOME=/usr/lib/jvm/java-1.8.0-openjdk-1.8.0.492.b09-2.el9.x86_64/jre
    export HADOOP_HOME=/usr/local/hadoop-3.3.6
    export HADOOP_CONF_DIR=$HOME/hadoop-configuration
    export PATH=$HADOOP_HOME/bin:$PATH
    export CLASSPATH=$(hadoop classpath --glob)

These are also documented in `.env` for reference.

### Deployment Steps (Impala mode)

    # 1. kinit to get a Kerberos ticket
    kinit <your-principal>

    # 2. Run pre-flight checks
    python MCP/hive_startup_check.py

    # 3. Enable Impala mode
    # In .env: HIVE_MCP_ENABLED=true

    # 4. Inject schemas + few-shots into Milvus (first time only)
    # Note: Ensure the uvicorn server is stopped first to avoid Milvus locking issues.
    python MCP/build_milvus_index.py
    python pipeline.py --config config.yaml --fewshots school_dropout_fewshots_200.jsonl

    # 5. Start the server
    python -m uvicorn app:app --host 0.0.0.0 --port 8000

## 5. SQLite vs Impala Configuration (`.env`)
The database backend is controlled via the `HIVE_MCP_ENABLED` environment variable:

*   **SQLite Mode (`HIVE_MCP_ENABLED=false`)**:
    *   **Registry**: Loads `MCP/mcp_sql_execution.py` exposing the `execute_sql` tool.
        Connects to `database/schema.db`.
    *   **Prompt**: The system prompt instructs the LLM that the database is SQLite and all
        queries must omit database prefixes (e.g. `SELECT * FROM citizen_student` instead of
        `SELECT * FROM curated_datamodels.citizen_student`).
    *   **RAG Preprocessing**: When `retrive_schema_rag` returns DDL, it strips out the
        `curated_datamodels.` prefix, removes Iceberg keywords (`USING ICEBERG`, location,
        table properties), and replaces Big Data types (`STRING` -> `TEXT`, `BIGINT` ->
        `INTEGER`) to prevent the LLM from writing invalid/unsupported syntax.

*   **Impala Mode (`HIVE_MCP_ENABLED=true`)**:
    *   **Registry**: Loads `MCP/mcp_hive_execution.py`. The tool inside is named `execute_sql`
        to maintain signature parity with the frontend/agent, but connects to Impala via Impyla.
    *   **Prompt**: The system prompt instructs the LLM that the database is Impala/Iceberg
        and queries must include table prefixes
        (e.g. `curated_datamodels.citizen_student`).
    *   **RAG Preprocessing**: Schema DDLs and table prefixes are returned exactly as-is from
        the vector index.

## 6. Agent Graph & Reasoning Nudges
The graph uses a standard loop: `START -> llm_node <-> tool_node -> END`.

The LLM determines whether it needs to fetch context (`retrive_schema_rag`) and when to execute
queries (`execute_sql`). To ensure reliability with smaller models (like `2b`), `llm_node` in
`my_agent/utils/nodes.py` implements reasoning nudges:
1.  **Nudge 1 (No Tool Call)**: If the user asks a database question but the LLM responds with a
    conversational answer on its first attempt, the agent appends a nudge message reminding it
    that it MUST use the tools, forcing a retry.
2.  **Nudge 2 (RAG but no SQL)**: If the LLM successfully calls the RAG tool but tries to
    describe the schema in text instead of running a SQL query, the agent appends a second nudge,
    instructing the model to immediately call `execute_sql`.

### User Query Lifecycle (Step-by-Step)
Here is exactly what happens from the moment a user submits a query (e.g., via `/ask` or `/api/chat` endpoints in `app.py`) to when the final response is returned:

1. **Initialization (`initialize_node`):**
   * The user query is placed into the LangGraph state.
   * The graph starts at `initialize_node` in [nodes.py](file:///Users/prakhar/Downloads/Student_Dropout_v1.0/my_agent/utils/nodes.py).
   * This node generates a forced/initial `AIMessage` containing a tool call to `retrive_schema_rag(query=user_query, top_k=5)`. This guarantees schema retrieval occurs before any SQL generation is attempted.

2. **RAG Tool Execution (`tool_node`):**
   * The `tool_node` intercepts the tool call and runs it using the RAG MCP server ([mcp_rag.py](file:///Users/prakhar/Downloads/Student_Dropout_v1.0/MCP/mcp_rag.py)).
   * The RAG server computes the query embedding (using Ollama's `nomic-embed-text`) and performs a semantic search against the Milvus Lite vector database (`milvus_schemas.db`).
   * It retrieves matching table schemas and the join relations chunk (from `join_relations.yaml`), processes/normalizes them (depending on SQLite vs Impala mode), and returns the results to the agent state as a `ToolMessage`.

3. **LLM Reasoning & SQL Generation (`llm_node`):**
   * The graph transitions to `llm_node`. The LLM receives the system prompt (defining core tables, instructions, and hardcoded main joins) along with the chat history (which now includes the retrieved RAG schema details).
   * The LLM writes the SQL query and outputs a tool call to `execute_sql(query=...)`.
   * *Nudges:* If the LLM generates a conversational response on its first turn instead of calling `execute_sql`, or if it tries to describe the schema in words, the agent appends a corrective nudge message and forces the LLM to invoke the tool.

4. **SQL Execution (`tool_node`):**
   * The `tool_node` executes `execute_sql` by communicating with the SQL execution MCP server:
     * Under SQLite mode (`HIVE_MCP_ENABLED=false`): Runs queries against the local `database/schema.db`.
     * Under Impala mode (`HIVE_MCP_ENABLED=true`): Runs queries via Impyla on the remote Impala instance.
   * The server returns the query results (JSON representation of columns and rows) or an execution error as a `ToolMessage`.

5. **SQL Result Verification (`verify_node`):**
   * The graph transitions to `verify_node`. The verifier LLM is called with a prompt containing:
     * The original user query.
     * The executed SQL query.
     * The first few rows of the query result.
   * **Verification Verdicts:**
     * **`RETRY`:** If the query execution failed or the verifier determines the result is incorrect/incomplete, the node injects a corrective human instruction message into the state and routes the graph back to `llm_node` to correct and re-run the SQL.
     * **`CORRECT`:** Once the verifier approves the result, the state variable `verified` is set to `True`, and the graph continues.

6. **Final Response Generation & Formatting (`llm_node`):**
   * After verification passes, the graph returns to `llm_node`.
   * It sees the successful SQL tool output and generates a clean, natural-language summary of the data (formatted nicely for the dashboard UI).
   * The graph transitions to the `END` state, and the final response is returned to the user.

*   `app.py`                       Dashboard routes, database seeding hooks, and agent REST API wrapper.
